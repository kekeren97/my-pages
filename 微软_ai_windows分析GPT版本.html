Operating model

微软正在构建的，不只是一个 AI 功能，而是一套四层结构

从硬件、运行时到 Copilot，再到应用层，微软正在把 AI 逐步塞进 Windows 的基础设施里，让“理解上下文、做出动作、调用软件”成为操作系统级能力。

应用层 Apps
Office、Outlook、Photos、Edge、第三方应用，继续承担具体功能与专业能力。
智能体层 Copilot
用户以自然语言表达意图，Copilot 负责理解、解释、组织与触发后续流程。
AI Runtime 层
Windows AI Runtime 承担模型调用、系统能力接入、推理路径分配等底层职责。
硬件层 CPU + GPU + NPU
通过 NPU 处理高频 AI 任务，把一部分原本依赖云端的推理成本转移到终端设备上。
Memory layer

Semantic Index：让电脑第一次拥有“长期记忆”

微软在 Recall 路线上最核心的突破，不是搜索框更聪明，而是把“用户行为历史”转成可被 AI 理解的语义索引。这样一来，AI 不再是一个不懂上下文的外部访客，而是逐步拥有与你工作流相关的记忆层。

01

它在做什么？

系统以时间序列记录用户的活动痕迹，把屏幕内容、文档片段、网页、聊天界面等信息转化为可检索的语义表示。 用户不再必须记住文件名、路径或关键词，而可以通过“模糊记忆”反向找回内容。

上周那个蓝色 PPT 我看过的东京酒店页面 昨天会议里提到的报价表
02

它为什么重要？

因为未来 AI 的价值不只取决于模型能力，更取决于是否掌握“与你相关的上下文”。如果没有记忆层，AI 只能回答通用问题； 一旦有语义索引，AI 才可能真正理解你的工作对象、历史轨迹与当前任务。

记忆层 上下文层 个性化检索
A

传统搜索

核心依赖关键词、文件名、路径和标签。你得先“记得”，系统才能帮你找到。

B

语义搜索

核心依赖 embedding 和向量检索。即使你只记得模糊特征，也能通过语义关联完成召回。

C

业务意义

它让 Windows 从“文件容器”变成“个人上下文数据库”，这是后续所有 Agent 能力的前提。

Hybrid AI

混合算力架构：不是所有任务都值得上云

微软的真正设计重点，不是“把最强模型装进电脑”，而是把任务分层：低延迟、高频、隐私敏感的任务尽量留在本地；复杂、多步骤、推理密集的任务再交给 Azure 云端模型。

On-device AI

本地 AI

  • 依赖 NPU 与小模型运行，强调低延迟、低成本、离线可用。
  • 适合 OCR、实时字幕、局部总结、翻译、图像理解、Recall 索引等任务。
  • 敏感数据可不离开设备，企业更容易接受。
Cloud AI

云端 AI

  • 依赖 Azure 模型完成长链路、多轮、复杂推理任务。
  • 适合长文本生成、代码、数据分析、多步骤工作流与高复杂度内容创作。
  • 承担能力上限，但不适合覆盖全部日常高频调用。
￥

原因一：算力成本

如果每一次摘要、搜索、理解都走云端，Windows 规模下的推理成本会非常高。把轻任务放到本地，是最现实的经济学选择。

⚡

原因二：响应速度

很多系统级体验必须是“瞬时”的，尤其是搜索、翻译、界面理解、摄像头与音频增强等场景，本地推理更符合体验要求。

🔒

原因三：隐私与合规

企业采购 AI 的第一道门槛不是模型够不够强，而是数据能否留在本地、权限边界是否清晰、是否可审计。

Action orchestration

跨应用编排：Copilot 不想只回答问题，它还想“替你调度软件”

微软对未来 Windows 的设想，不是让 AI 永远停留在聊天框里，而是让它逐步具备跨应用理解与调用能力。真正的变化不是“生成答案”，而是“把答案落成动作”。

意

Strategic logic

微软为什么必须这样做？背后不是技术浪漫，而是平台保卫战

从商业逻辑看，这套路线并不神秘：它同时解决 Windows 的入口危机、AI 成本问题、企业隐私问题，以及 PC 行业新的升级叙事。

1

防止 Windows 被边缘化

如果 AI 只存在于浏览器网页，用户对操作系统的依赖会迅速下降。微软必须让“系统级 AI 能力”成为 Windows 独有的价值锚点。

2

保住平台控制权

未来的软件入口不再只是开始菜单和桌面图标，而是自然语言与 Agent。谁掌握这个入口，谁就重新掌握生态调度权。

3

推动硬件换代

Copilot+ PC 与 NPU 标准，把 AI 从“云服务故事”重新拉回到“终端设备升级逻辑”，这对微软和 OEM 都极其关键。

4

企业市场需要“本地可控”

政企客户采购 AI，核心顾虑并不是“模型参数不够大”，而是数据有没有离开本地、系统能否做权限分层、审计能否闭环。 微软把 AI 下沉到 Windows 设备端，本质上是在构建企业级信任基础设施。

5

让 AI 从工具升级为基础设施

单独的聊天产品容易被替代，但操作系统级的记忆、权限、设备调用、软件编排能力更难复制。 微软不是只想做一个回答问题的 AI，而是想占住“数字工作入口”的新底座。

Evolution

这件事放到更长的时间线里看，本质上是两次 Windows 设计范式切换

第一次，Windows 解决的是“普通人如何通过图形界面使用电脑”；第二次，微软想解决的是“电脑如何通过自然语言理解人的意图”。

1995 之后
GUI 时代：人学习软件

用户必须理解菜单、窗口、图标、路径、格式与功能入口。软件是工具箱，学习成本在用户身上。

云与移动时代
平台时代：入口开始漂移

浏览器、手机、SaaS 和云服务逐渐削弱了操作系统的中心地位，Windows 的入口优势开始被稀释。

2024–2026
Copilot+ PC 时代：AI 被嵌入系统层

微软开始用 Recall、本地 NPU、AI Runtime 和 Copilot，把“理解上下文 + 执行动作”的能力往底层收回。

下一阶段
Agent OS：软件开始理解人

未来最重要的不再是哪个 App 图标更强，而是谁能掌握上下文、权限、动作编排与可持续记忆层。

Takeaway

真正值得记住的结论，不是“微软加了很多 AI 功能”

而是微软正在尝试把 Windows 从一个应用承载平台，改造成一个具备记忆、推理、动作与上下文整合能力的智能系统底座。

One-line summary
过去是“人学习软件”，未来是“软件理解人”。
对微软而言，这不是一次普通功能升级，而是一次平台重定义： 让 Windows 不只是运行软件，而是理解意图、连接上下文、编排动作，并重新成为 AI 时代最重要的终端入口之一。
