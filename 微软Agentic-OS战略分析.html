Section 01

微软到底在做什么？

微软没有使用过"AI-Native Windows"这个说法。它真正在推的概念，官方用语叫 Agentic OS——把 Windows 从一个用户手动操作应用的平台，变成一个 AI agent 可以自主完成任务的平台。官方

在 2025 年 11 月的 Ignite 大会上，Windows 负责人 Pavan Davuluri 公开使用了这一表述。内部相关的代号是 "Hudson Valley"（下一代 Windows 重大更新）和 "CorePC"（模块化架构方向）。但截至目前，微软官方从未确认过"Windows 12"的存在，公开策略仍然是持续迭代 Windows 11。可验证

"We are now entering a phase where we build rich scaffolds that orchestrate multiple models and agents; account for memory and entitlements; enable rich and safe tools use." —— Satya Nadella，微软 CEO，2025 年末展望博文

简单说，微软的野心不是"给 Windows 加一个 AI 助手"，而是让 Windows 本身变成 AI agent 的运行环境。类比来说：1990 年代 Windows 3.0 的胜利不是靠 GUI 本身，而是靠让 Windows 成为第三方软件的"平台"。现在微软想重演这个故事——让 Windows 成为 AI agent 的平台。

三个已落地（或在落地中）的技术模块

语义索引 Semantic Index

让系统"理解"你的文件，而不只是匹配关键词

Windows 在本地建立向量索引，理解文件内容的语义含义。你搜"上周那个关于预算的表格"，即使文件名叫 MKT_2025_Plan.xlsx，系统也能找到。官方

推送时间
2025年5月 Windows 11 安全更新起
硬件要求
Copilot+ PC，NPU ≥ 40 TOPS
支持语言
中文（简体）、英、法、德、日、西
数据处理
完全本地，微软声明不收集/不训练

支持的文件格式：

.txt.pdf.docx.doc.rtf.pptx.ppt.xls.xlsx.jpg.jpeg.png.gif.bmp.ico
常见误解纠正
语义索引 ≠ Recall。Recall 是通过定期截屏记录用户在屏幕上看到的内容，创建可搜索的时间线。语义索引是对文件内容本身建立向量化的语义理解。两者都在本地运行，但数据来源和用途完全不同。
两套语义索引的区别
Windows 本地的语义索引处理设备上的文件；M365 Copilot 的语义索引处理的是 Microsoft Graph 里的组织数据（邮件、Teams 聊天、SharePoint 等），走的是云端。技术原理相似，作用范围和部署位置完全不同。

本地 AI 推理：NPU + Phi Silica

在设备上运行小语言模型，无需联网

微软在 Copilot+ PC 的操作系统中内置了 Phi Silica——一个专门为 NPU 优化的小语言模型（SLM），属于 Phi 系列模型的"姐妹版"。它可以在不联网的情况下执行轻量 AI 任务（文本摘要、本地搜索建议、UI 提示生成等）。官方

模型优化
QuaRot 4-bit 量化，NPU 专用调优
功耗表现
上下文处理仅 4.8mWh（Snapdragon X）
硬件覆盖
Qualcomm / Intel / AMD 三平台
更新方式
Windows Update 独立组件，不绑定 OS 大版本

数据来源：Windows Experience Blog（2024.12）、KB5079266 等更新记录

"混合算力"是怎么工作的？

微软确实在走本地 + 云端的混合路线，但具体的路由机制没有公开的标准化文档。分析推断

轻量任务 → 本地
文本摘要、本地搜索建议、UI 提示生成等，由 Phi Silica 在 NPU 上完成
重度任务 → 云端
涉及跨租户数据、M365 集成、需要更强推理能力的任务，调用云端大模型
开发者可自定义路由
根据网络状态、隐私敏感度、电池状态等条件，在本地 ONNX 模型和 Azure OpenAI 之间切换
关键澄清
这不是一个自动的、透明的"无缝切换"系统。它更像是一套框架和工具，具体切换逻辑取决于应用开发者怎么实现。"80% 任务在本地完成"之类的说法没有官方出处。
⚡

Agent 架构：让 AI 在系统层面"做事"

Agentic OS 概念的核心——AI 不只聊天，还能操作

这是微软 2025 年末以来推得最猛的方向。核心思路：让 AI agent 能够在操作系统级别发现工具、执行工作流、跨应用交互——而不只是在侧边栏回答问题。官方

Agent Workspace 代理工作区

独立的安全沙箱，agent 拥有自己的账户（与用户隔离），在后台执行任务。用户可随时查看行为并撤销权限。

Agent Launchers 代理启动器

开发者通过 manifest 注册 agent，注册后可出现在任务栏、Copilot 界面和其他应用中。已在 Insider 预览。

MCP 协议支持

采用 Anthropic 创建的开放协议 Model Context Protocol，作为 agent 与工具交互的标准接口。非专有方案。

企业管理集成

通过 Intune、Entra、Group Policy 管理 agentic 功能。包括启用/禁用 connector、设定安全策略等。

与常见叙述的差异
很多文章说 Copilot "直接调用 Windows API 操作应用"——这是早期的做法。现在微软走的是更开放的 agent connector + MCP 协议路线，第三方（包括 Anthropic Claude）也可以接入。可验证
Section 03

微软为什么要这么做？

以下三点是基于公开信息的 分析推断，不是微软官方说法，但有充分的事实支撑。

推断 01

算力成本决定了必须做本地推理

Windows 全球活跃设备超过 10 亿台（微软 FY2026 Q2 财报确认 Windows 11 已超 10 亿台）。如果每个 Copilot 交互都走云端推理，Azure 的计算成本将不可持续。

把轻量交互下沉到本地 NPU + 小模型，是唯一在经济上可行的路径。这也解释了为什么微软要定义 Copilot+ PC 的硬件门槛（40 TOPS NPU）——它需要硬件厂商帮它分担推理算力的基础设施成本。本质上是把推理成本转嫁到硬件销售环节：用户买了 Copilot+ PC，就等于预付了本地 AI 的"算力费"。

推断 02

本地处理是打开政企市场的钥匙

政企客户对云端 AI 处理敏感数据极度警惕。微软在 Recall 功能上的隐私争议就是一个教训——2024 年 Recall 首次发布时因隐私安全问题被迫延期整整一年。

把 AI 推理锁定在本地 NPU 闭环，配合 Purview（数据防泄露）、Intune（设备管理）等企业工具，微软可以构建一套"AI 能力强但数据不出设备"的叙事——这对合规敏感的行业（金融、国防、政府）至关重要。

推断 03

防止 Windows 被 AI 浏览器化架空

如果 AI 的主要入口是浏览器里的 ChatGPT / Gemini / Claude 网页版，那操作系统就变成了一个无差异的"容器"——用什么 OS 都无所谓。

微软必须让 AI 与 Windows 深度绑定：agent 可以直接操作本地文件、调用系统设置、在后台跨应用编排任务——这些是网页版 AI 做不到的。这是在用 AI 重新定义操作系统的不可替代性。

Section 04

现实中的阻力和质疑

愿景宏大，但推进中问题不少。只讲愿景不讲阻力的分析，对判断没什么帮助。

🔴

用户反弹已经发生

2025 年，微软在 Notepad、Paint、File Explorer 等系统应用中大量塞入 Copilot 按钮，被广泛批评为"侵入式"且缺乏实际价值。Davuluri 在 X 上发布 Agentic OS 愿景时收到大量负面回复，被迫关闭评论。可验证

🟡

微软已承认过度扩张

2026 年初，Davuluri 接受采访时承认了用户的"痛点"，启动了内部称为"swarming"的策略——把工程资源从新功能开发重新调配到可靠性和性能修复上。部分 Copilot 集成正在被审查，可能移除或重新设计。可验证

🟠

硬件门槛造成市场分裂

40 TOPS NPU 的要求意味着绝大多数现有 PC 无法使用这些 AI 功能。这不是"渐进升级"，而是一道硬件墙。Copilot+ PC 的市场渗透率仍然很低，大量企业客户还在 Windows 10 上。分析推断

⚪

Agent 的实际能力还很初期

Agent Workspace 目前仅在 Insider 预览版中提供，首批支持的只有 Copilot Actions，第三方 agent 生态几乎为零。从"AI 能调系统设置"到"AI 能自动编排多应用完成复杂工作流"，中间还有巨大的工程鸿沟。可验证

Section 06

对国产办公软件厂商的启示

分析推断 以下为独立分析观点

启示 01

平台控制力的重新分配

微软的"本地 AI + 操作系统绑定"路线，强化了它作为平台方的控制力。如果 AI agent 必须通过 Windows API / MCP 协议才能操作应用，那么不在微软生态内的办公软件天然处于劣势。国产办公软件需要思考：在 Windows 的 agentic 框架下，自己是被编排的对象，还是有机会成为编排者？

启示 02

本地小模型路线的商业验证

Phi Silica + NPU 的路线验证了一个判断：AI 商业化不一定要靠大模型。对于私有化部署场景，客户关心的不是模型有多大，而是"能不能在我的环境里跑、数据出不出去"。微软正在用操作系统级别的方案回应这个需求，国产办公软件需要在应用层面给出同样有说服力的回答。

启示 03

信任建设先于能力释放

微软的 Recall 隐私风波是前车之鉴。任何涉及"AI 扫描用户内容"的功能，无论技术上多安全，都会面临信任挑战。先建立信任，再释放能力，可能比"先上线再解释"更可行——这一点在中国的政企市场尤其如此。
